🐳 Sobre “Docker complexo” (o problema não é usar Docker)

Vou ser direto:
👉 usar Docker = bom
👉 exagerar no Docker = risco

✅ Quando Docker vira diferencial

Se você fizer algo assim:

1 container backend (FastAPI)
1 container frontend (Next.js)
1 container banco (PostgreSQL)
docker-compose simples

👉 Isso é excelente
Mostra:

maturidade
reprodutibilidade
facilidade de avaliação
❌ Quando Docker começa a te atrapalhar

O problema é quando vira isso:

múltiplos Dockerfiles desnecessários
nginx configurado na mão
rede customizada complexa
scripts de entrypoint complicados
problemas de build/debug

👉 Aí você:

perde tempo
aumenta chance de erro
dificulta correção em cima da hora
🎯 Regra prática (importante)

Se o avaliador precisar de mais de 3 comandos pra rodar, você já começou a perder pontos.

✅ O que eu recomendo (melhor dos dois mundos)

Use Docker, mas assim:

version: "3.9"

services:
  backend:
    build: ./backend
    ports:
      - "8000:8000"
    depends_on:
      - db

  frontend:
    build: ./frontend
    ports:
      - "3000:3000"

  db:
    image: postgres:15
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: password
      POSTGRES_DB: weather
    ports:
      - "5432:5432"

E no README:

docker-compose up --build

👉 Pronto. Isso já impressiona.

🧠 Insight importante

Recrutador não pensa:

"Nossa, que arquitetura incrível com Docker"

Ele pensa:

"Consigo rodar isso fácil?"
"Está organizado?"
"Esse cara sabe priorizar?"
